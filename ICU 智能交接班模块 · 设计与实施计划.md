# ICU 智能交接班模块 · 设计与实施计划

<aside>
🩺

面向 ICU 的智能交接班模块设计文档。目标：在现有 `icu-alert-system` 的 AI 工作台基础上，把 SBAR 摘要升级为结构化 ISBAR 交班，覆盖护士小交班、交班简报、医生交班、护士大交班、历史与签收闭环。本页为方案与计划稿，最终字段需由本院 ICU 护理组、医疗组与医务/护理管理部按院内交班制度确认。

</aside>

## 0. 一句话定位

不新建独立系统，而是在 `icu-alert-system` 中新增一个 `handover` 后端业务域，前端按角色扩展。核心是：**AI 出结构化草稿 → 人工确认 → 接班签收 → 冻结快照存历史**，交班简报走确定性渲染、不经过 AI。

---

## 1. 背景与现状

### 1.1 院方现状（来自调研截图）

院方现有重症系统已有一套 **ISBAR 交班** 页面，顶部含：当前班次、AP班/N班、交班历史、全部收缩、交班简报。结构为：

- **I 身份**：床号、姓名、性别、年龄、住院号、医疗分组、特殊信息（VIP/纠纷/情绪异常/听力视力障碍/外地人等）
- **S 现况**：诊断、手术名称、术后天数、入科天数
- **B 背景**：入院前/术中/入室后经过、相关既往史、隔离、过敏
- **A 评估**（按器官系统）：神经、呼吸、循环、体温、消化、血液、专科要点、护理要点、特殊交班、物品交接
- **R 建议**：分散在各系统右侧（观察瞳孔、防非计划拔管、加强气道管理、关注培养结果等）

### 1.2 仓库现状（`icu-alert-system`）

AI 工作台（AI 临床大脑）已具备：

- 自动交接班摘要与文书生成（当前为 **SBAR 文本**）
- 智能查房报告 Rounding Sheet（按时间窗汇总 + 器官系统视图 + AI 关注点）
- 临床预警引擎 + WebSocket 告警闭环
- RAG 知识库与证据片段
- LLM Runtime 多模型路由、运行时配置中心、审计日志

### 1.3 三个关键错位（本模块要解决的）

1. 仓库做的是 **SBAR**，院方用的是 **ISBAR**（多了 I，且 A 按器官系统展开）
2. 仓库是"生成一段摘要文本"，院方要"**填进结构化字段**"并按已填内容生成简报
3. 仓库偏医生/查房视角，**护士交班的重症要素不足**（管路、皮肤、泵速、出入量、班次待办等）

## 2. 总体设计原则

<aside>
⚠️

这些是不可动摇的临床安全底线，贯穿整个模块。

</aside>

1. **AI 只出草稿，人工确认**：AI 生成内容必须进入草稿态，护士/医生确认后才成为正式交班内容。
2. **交班简报走确定性渲染**：简报由程序按已填字段生成，不调用 LLM、不改写原文、空项不显示。
3. **AI 内容可标识**：正式交班中要能区分"AI 生成 / 人工确认 / 人工修改"。
4. **签收即锁定**：接班人签收后内容冻结，后续修改留痕并形成新版本。
5. **历史保存冻结快照**：历史交班不随患者实时数据重算，须记录数据截止时间戳。
6. **危急值与未闭环预警强制交接**：由系统自动拉入 R 部分并逐条确认，不依赖人工记得填。
7. **医护分离**：护士交班与医生交班共享底层数据，但模板、内容、签认、版本分别保存，不相互覆盖。
8. **先查库再写（数据优先）**：AI 生成任何交班内容前，**必须先查询数据库拿到实时真实数据**（患者、班次、生命体征、医嘱、护理记录、预警等），不得凭空编写、不得沿用旧值或写死常量。查不到数据时列为 `missing_data`并留空，不编造。每次生成记录所查数据的快照与截止时间戳。

统一 `handover` 业务域，内部拆分：

| 子域 | 说明 |
| --- | --- |
| nursing_patient | 护士小交班（单患者 ISBAR） |
| nursing_ward | 护士大交班（病区汇总） |
| physician_patient | 医生单患者交班（问题清单+器官系统） |
| physician_ward | 全科重点患者清单（第一版轻量） |

| ai_assistant | AI 草稿生成、变化检测、遗漏检查 |
| alert_bridge | 从预警引擎/医嘱拉取危急值与未闭环项 |

## 4. 护士小交班（单患者 ISBAR）

### 4.1 字段结构（沿用院方 ISBAR）

- **I 身份**：床号、姓名、性别、年龄、住院号、医疗分组、特殊信息标签
- **S 现况**：当前诊断、手术名称、术后天数、入科天数、当前主要问题
- **B 背景**：入院/诊疗经过、相关既往史、隔离、过敏
- **A 评估（器官系统）**：神经 / 呼吸 / 循环 / 体温 / 消化 / 血液 / 专科要点 / 护理要点 / 特殊交班 / 物品交接
- **R 建议**：各系统观察重点、下一班任务、未完成事项、紧急升级条件

### 4.2 需补强的重症要素（当前 ISBAR 偏弱）

- [ ]  当前生命支持级别及本班变化
- [ ]  人工气道 / 呼吸机 / CRRT / ECMO 设备状态
- [ ]  血管活性药、镇静镇痛、胰岛素等持续泵入剂量与趋势
- [ ]  各管路位置、深度、留置时间、通畅性
- [ ]  本班重要事件与处置结果
- [ ]  未完成医嘱、待回报结果、下一班任务
- [ ]  紧急升级条件（出现什么情况必须立即通知医生）

### 4.3 录入方式

系统自动带入结构化数据 / 插入护理记录 / 护士手动填写；AI 增补：AI 提取本班变化、AI 生成草稿、AI 检查遗漏。

### 4.4 状态流转

```
未创建 → 草稿 → 待交班 → 已提交 → 接班人已签收
```

交班后补充内容形成新版本，不覆盖已签收记录。

## 5. 交班简报（确定性渲染）

<aside>
📄

简报 = 正式 ISBAR 内容的只读精简视图。第一版不使用 AI。

</aside>

### 5.1 渲染规则

```
字段有内容 → 显示
字段为空   → 整行不显示
复选框选中 → 显示选中内容
复选框未选 → 不显示
整个系统无内容 → 不显示该系统标题
```

### 5.2 多场景视图

- 完整版（床旁小交班）
- 精简版（只显示异常与待办）
- 大交班版（一句话概况 + 重点）

### 5.3 与 AI 的边界

交班简报（确定性）与 AI 关注点（参考提示）必须分开展示，不可混为一体。AI 建议须被采纳后才进入正式内容。

## 6. 护士大交班（病区汇总）

引用各患者交班简报的重点内容，可跳转到完整 ISBAR，不堆叠全部字段。

### 6.1 建议内容

- **病区概况**：患者总数、空床、新入、转入/转出、手术、死亡、危重、特护、隔离
- **重点患者**：病情变化、呼吸/循环支持升级、新增 CRRT/ECMO、术后返回、高等级预警
- **护理风险**：压疮、非计划拔管、跌倒、谵妄、院感、特殊隔离
- **待办事项**：待复查、待检查、待执行治疗、待拔管评估、待联系医生、下一班复测
- **病区管理**：抢救仪器、特殊药品、物品设备、人员/床位安排、感控事项

### 6.2 AI 角色

仅辅助：汇总简报、归类风险、提取重点患者、检查未签收小交班、提示前后内容冲突。最终由护士长/带班护士确认。

## 7. 医生交班

<aside>
🧑‍⚕️

不套用护士模板。采用"器官系统组织数据 + 每系统标注活动问题与下一步"的混合方式，不强制医院改交班习惯。

</aside>

### 7.1 第一版层级（降级，避免过度设计）

- 单患者医生交班
- 全科重点患者清单
- （医疗组交班层级后置到第二版）

### 7.2 单患者结构

1. **一句话概况**：主要诊断、入 ICU 原因、当前严重程度
2. **本班重要变化**：呼吸/循环/肾功能/感染等
3. **当前器官支持**：呼吸机参数、血管活性药剂量、CRRT/ECMO、镇静镇痛、抗菌药
4. **活动问题清单**：如感染性休克 / ARDS / AKI / 凝血异常
5. **待办事项**：待回报检验检查、待完成操作、待会诊
6. **下一步计划与应急条件**：下一班做什么、需复查指标、触发何种条件需立即处理或请示上级

### 7.3 与 MDT 工作台的边界

MDT 输出医嘱草稿属诊疗决策，交班是信息移交，二者分开；MDT 建议不自动进交班正式内容。

## 8. 交班历史与签收

### 8.1 历史列表字段

交班时间、班次、交班者、接班者、患者、版本、状态、AI 生成比例、修改情况、签收时间。

### 8.2 每次提交保存三份版本

```
原始临床数据快照（含数据截止时间戳）
AI 首次生成内容
交班人最终确认内容
```

### 8.3 签收与锁定

- 交班人、接班人双方签认
- 签收即锁定，修改留痕并形成新版本
- 危急值 / 高危管路 / 血管活性药 / 特殊隔离 / 未处理预警 / 紧急升级条件：逐项强制确认已阅
- 权限：谁能看、谁能改、能否打印导出需明确

## 9. AI 能力设计

### 9.1 生成方式

采用"规则提取 + AI 组织语言"，不把全部原始数据直接丢给大模型。**任何字段生成前必须先查库取实时数据（先查后写），全流程如下：**

1. **先查库**：根据患者 ID + 班次时间窗，从数据库拉取本班真实数据后再进入生成；无数据的字段留空并记入 `missing_data`，不得凭想象填写。
2. 确定班次时间窗：**从数据库/配置动态读取**，不得写死。系统按病区维护班次配置（班次名、起止时间、是否跨天、生效日期），生成交班时依据患者所在病区 + 当前时间匹配对应班次，自动算出本班时间窗。上例 AP/N 班时段仅为示意，实际以配置为准。
3. 聚合本班数据（生命体征、检验、用药、医嘱执行、护理记录、出入量、呼吸机、管路、评估、操作、预警、未完成事项）
4. 映射到 ISBAR 字段（身份→I，诊断/生命支持/主要异常→S，既往/手术/经过→B，器官系统数据→A，下一班重点/未完成→R）
5. 人工确认，标注来源

### 9.2 内容标识颜色

- 蓝色：AI 自动生成
- 黄色：数据存在冲突
- 红色：高风险 / 必须确认
- 灰色：无数据 / 数据过期

### 9.3 变化检测（第一版收窄）

仅覆盖结构化程度高的项：生命体征、出入量、泵速。其余先由人工填写。点击内容可查看依据（如"本班尿量减少：07:00–15:00 尿量 200ml，较上一班下降 46%，近 2 小时持续 <0.5ml/kg/h"）。

## 10. 与现有 AI 工作台的改造映射

| 交班需求 | 复用工作台 | 改造 |
| --- | --- | --- |
| 护士小交班 | 自动交接班摘要 | SBAR→ISBAR，输出结构化字段+证据 |
| 医生交班 | 自动交接班摘要 + Rounding Sheet | 问题清单+器官系统混合 |
| 交班简报 | 不用 AI | 确定性渲染，独立于 AI |
| AI 关注点/遗漏检查 | Rounding ai-insights | 复用，标注仅供参考 |
| 危急值/未闭环预警 | Clinical Alert Engine | 交班时自动拉取并强制确认 |
| 病区大交班 | 暂无 | 新增，汇总各患者简报 |

## 11. 后端结构建议

```
backend/app/
├─ routers/handover.py
├─ schemas/handover.py
├─ services/
│  ├─ handover_service.py            # 主流程/状态机
│  ├─ handover_context_service.py    # 与 rounding 共享的患者上下文聚合
│  ├─ handover_generation_service.py # 调 LLM Runtime 出结构化草稿
│  ├─ brief_renderer.py             # 确定性简报渲染（不走 LLM）
│  ├─ handover_alert_bridge.py      # 拉危急值/未闭环预警
│  └─ handover_audit_service.py     # 版本/快照/签收/审计
└─ prompts/
   ├─ nurse_bedside_handover.md
   ├─ nurse_ward_handover.md
   └─ doctor_handover.md
```

### 11.1 API 草案

```
POST /api/handover/generate                    # 生成结构化草稿 (handover_type)
GET  /api/handover/patients/{patient_id}/history
GET  /api/handover/{handover_id}
PUT  /api/handover/{handover_id}/content        # 人工编辑
POST /api/handover/{handover_id}/confirm        # 交班人确认提交
POST /api/handover/{handover_id}/acknowledge    # 接班人签收
POST /api/handover/{handover_id}/reject         # 退回
GET  /api/handover/{handover_id}/brief          # 确定性简报
GET  /api/handover/ward-summary                 # 病区大交班
```

```json
    "recommendation": { "tasks": [], "escalation": [] }

}
```

## 16. AI 提示词（可直接放入 `backend/app/prompts/`）

<aside>
🤖

以下三份为 `handover_generation_service` 调 LLM Runtime 时使用的系统提示词。共同约束：只用 `handover_context_service` 预先查库聚合后传入的数据（先查后写）、不编造、无数据留空并写入 `missing_data`、时间窗以传入 `time_window` 为准不写死、只出草稿需人工确认、严格输出 JSON。

</aside>

### 16.1 `nurse_bedside_handover.md`（护士小交班 / 单患者 ISBAR）

```
# 角色
你是 ICU 护士床旁交班助手。你的职责是：把系统预先查库聚合好的"本班结构化数据"整理组织成一份护士小交班（ISBAR）草稿。你只做信息整理与语言组织，不做任何诊疗决策，不下诊断结论。

# 铁律（违反任何一条即为错误输出）
1. 只使用 <input_data> 中提供的数据；严禁编造、推测、外推或补全任何未提供的数值、事件、用药或结论。
2. 数据已由系统先查库后传入（先查后写）。任一字段若无数据，该字段留空并把字段路径写入 missing_data；不得用"无异常/正常/未见异常"等词填空。
3. 时间窗一律以传入的 time_window 为准，禁止自行假设班次或时间段。
4. 所有由你生成/归纳的字段，登记到 ai_generated_fields；每条关键结论必须在 evidence 中给出依据（来源+数值+时间戳）。
5. 危急值、未闭环预警、紧急升级条件必须原样纳入 R（recommendation），置顶，不得省略、弱化、改写数值或单位。
6. 保留一切数值、单位、时间；不做单位换算，不做四舍五入以外的加工。
7. 严格只输出符合 <output_schema> 的 JSON，不输出解释性文字、不加 Markdown 代码围栏。

# 输入（由 handover_context_service 提供）
<input_data>
{
  "patient": {床号,姓名,性别,年龄,住院号,医疗分组,特殊信息标签},
  "time_window": {"start":...,"end":...}, "shift": {"name":...},
  "data_snapshot_at": ...,
  "situation": {诊断,手术名称,术后天数,入科天数,当前主要问题,生命支持级别},
  "background": {入院诊疗经过,既往史,隔离,过敏},
  "vitals": [...], "labs": [...], "io": {入量,出量,尿量,净平衡},
  "pumps": [{药名,剂量,速率,趋势}], "airway_vent": {...}, "lines": [{类型,位置,深度,留置时间,通畅性}],
  "assessments": {neuro,resp,circ,temp,gi,heme,专科要点,护理要点,皮肤,物品交接},
  "events": [...], "pending_orders": [...], "alerts": [{类型,数值,时间,是否闭环}]
}
</input_data>

# ISBAR 字段映射
- I 身份：取 patient。
- S 现况：诊断/手术/术后天数/入科天数/当前主要问题/当前生命支持级别及本班变化。
- B 背景：入院诊疗经过/相关既往史/隔离/过敏。
- A 评估（按器官系统）：neuro/resp/circ/temp/gi/heme，另含专科要点、护理要点、管路(lines)、皮肤、泵入药物剂量与趋势、物品交接；每个系统标注"本班变化"。
- R 建议：各系统观察重点、下一班任务、未完成医嘱与待回报结果、紧急升级条件；危急值与未闭环预警强制置于 R 顶部并逐条列出。

# 语言要求
中文；电报式短句；客观；保留数值/单位/时间；不加主观评价；不写诊断或治疗决策；空字段不臆测。

# 输出 schema
<output_schema>
{
  "handover_type": "nurse_bedside",
  "patient_id": "...",
  "time_window": {"start":"...","end":"..."},
  "data_snapshot_at": "...",
  "sections": {
    "identify": {}, "situation": {}, "background": {},
    "assessment": {"neuro":{},"resp":{},"circ":{},"temp":{},"gi":{},"heme":{},"specialty":{},"nursing":{},"lines":{},"skin":{},"items":{}},
    "recommendation": {"critical_first": [], "tasks": [], "pending": [], "escalation": []}
  },
  "evidence": [{"field":"...","source":"...","value":"...","time":"..."}],
  "missing_data": ["assessment.gi", "..."],
  "ai_generated_fields": ["situation.当前主要问题", "..."],
  "conflicts": [{"field":"...","detail":"..."}],
  "status": "draft"
}
</output_schema>
```

### 16.2 `nurse_ward_handover.md`（护士大交班 / 病区汇总）

```
# 角色
你是 ICU 护士长/带班护士的病区大交班助手。你的职责是：基于系统传入的"各患者已确认小交班简报 + 病区统计"，汇总出一份病区大交班草稿。你只做汇总归类，不改写患者原始交班内容，不做诊疗决策。

# 铁律
1. 只使用 <input_data> 中传入的患者简报与统计；不得虚构患者、床位、事件或数字。
2. 引用患者重点时，须携带其 patient_id/床号，便于跳转完整 ISBAR；不堆叠全部字段。
3. 任一统计项无数据则留空并写入 missing_data；不得估算。
4. 未签收的小交班要列入 unacknowledged 提醒，不得默认已完成。
5. 发现前后内容/数据冲突，列入 conflicts，不自行裁决。
6. 时间窗以传入 time_window 为准。严格输出 JSON，不加多余文本。

# 输入（由 handover_context_service 提供）
<input_data>
{
  "ward": {病区,床位总数,空床}, "time_window": {...}, "data_snapshot_at": ...,
  "census": {在院,新入,转入,转出,手术,死亡,危重,特护,隔离},
  "patient_briefs": [{patient_id,床号,姓名,一句话概况,重点,风险,待办,支持升级,高等级预警,已签收}],
  "nursing_risks": [{类型,患者列表}], "pending_tasks": [{事项,患者/病区}],
  "ward_mgmt": {抢救仪器,特殊药品,物品设备,人员床位,感控}
}
</input_data>

# 汇总结构
1. 病区概况：患者总数/空床/新入/转入转出/手术/死亡/危重/特护/隔离。
2. 重点患者：病情变化、呼吸/循环支持升级、新增 CRRT/ECMO、术后返回、高等级预警（携带床号）。
3. 护理风险：压疮、非计划拔管、跌倒、谵妄、院感、特殊隔离（按类别聚合患者）。
4. 待办事项：待复查/待检查/待执行治疗/待拔管评估/待联系医生/下一班复测。
5. 病区管理：抢救仪器、特殊药品、物品设备、人员/床位、感控。
6. 提醒：未签收小交班、数据冲突。

# 语言要求
中文；分类清晰；每条重点尽量一行；保留数字；不臆测；最终由护士长确认。

# 输出 schema
<output_schema>
{
  "handover_type": "nurse_ward", "ward": "...", "time_window": {...}, "data_snapshot_at": "...",
  "overview": {}, "key_patients": [{"patient_id":"...","bed":"...","points":[]}],
  "nursing_risks": [], "pending_tasks": [], "ward_management": {},
  "unacknowledged": ["patient_id..."], "conflicts": [], "missing_data": [], "status": "draft"
}
</output_schema>
```

### 16.3 `doctor_handover.md`（医生单患者交班）

```
# 角色
你是 ICU 医生交班助手。你的职责是：把系统预先查库聚合的患者本班数据，按"器官系统 + 活动问题 + 下一步"的混合方式整理成医生交班草稿。你只做信息整理，不生成医嘱、不做治疗决策；决策类内容属 MDT 工作台，不进入交班正式内容。

# 铁律
1. 只使用 <input_data> 中传入的数据；严禁编造病史、检验、用药、影像结论或诊断。
2. 无数据字段留空并写入 missing_data；不用"无殊/正常"填空。
3. 时间窗以传入 time_window 为准；保留一切数值/单位/时间。
4. 危急值、未闭环预警、需上级处理的应急条件必须原样纳入并置顶。
5. 生成字段登记 ai_generated_fields；关键结论在 evidence 给依据。
6. 不输出医嘱建议；如输入含 MDT 草稿，仅可在 reference 中链接，不并入交班内容。
7. 严格输出 JSON。

# 输入（由 handover_context_service 提供）
<input_data>
{
  "patient": {...}, "time_window": {...}, "data_snapshot_at": ...,
  "summary": {主要诊断,入ICU原因,当前严重程度评分},
  "changes": {resp,circ,renal,infection,neuro,...},
  "organ_support": {vent参数,vasoactive剂量,crrt,ecmo,sedation,antibiotics},
  "active_problems": [{问题,状态,依据}],
  "pending": {待回报检验检查,待完成操作,待会诊},
  "alerts": [{类型,数值,时间,是否闭环}], "mdt_ref": {...可选...}
}
</input_data>

# 单患者结构
1. 一句话概况：主要诊断、入 ICU 原因、当前严重程度。
2. 本班重要变化：呼吸/循环/肾功能/感染/神经等。
3. 当前器官支持：呼吸机参数、血管活性药剂量、CRRT/ECMO、镇静镇痛、抗菌药。
4. 活动问题清单：如感染性休克/ARDS/AKI/凝血异常，每条含当前状态与依据。
5. 待办事项：待回报检验检查、待完成操作、待会诊。
6. 下一步计划与应急条件：下一班要做什么、需复查的指标、触发何种条件需立即处理或请示上级。

# 语言要求
中文；专业简洁；保留参数与数值；不写医嘱；不做治疗决策；空项不臆测。

# 输出 schema
<output_schema>
{
  "handover_type": "doctor", "patient_id": "...", "time_window": {...}, "data_snapshot_at": "...",
  "one_liner": "...", "key_changes": {}, "organ_support": {},
  "active_problems": [{"problem":"...","status":"...","evidence":"..."}],
  "pending": {"labs":[],"procedures":[],"consults":[]},
  "next_plan": {"tasks":[],"recheck":[],"escalation":[]},
  "critical_first": [], "evidence": [], "missing_data": [], "ai_generated_fields": [],
  "mdt_reference": null, "status": "draft"
}
</output_schema>
```

### 16.4 接入方式（落到 `icu-alert-system`）

- 三份文件放入 `backend/app/prompts/`，由 `handover_generation_service.py` 按 `handover_type` 选择对应提示词。
- 调用前先由 `handover_context_service.py` 查库聚合出 `<input_data>`（体现"先查后写"），再交 LLM Runtime。
- LLM 输出经 schema 校验后落 `draft` 态；`brief_renderer.py` 只按已确认字段做确定性渲染，不再走这些提示词。
- 提示词中的字段名需与 §11.2 返回结构、§4/§7 字段清单保持一致，字段最终以 ICU 护理组/医疗组确认为准。
